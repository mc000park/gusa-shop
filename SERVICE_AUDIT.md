> 대상: GUSA 교재 전문 쇼핑몰 (apex/ + gusaFront/)

|------|--------|-----------|------------|

## 1. 완전히 구현된 기능 ✅

### 회원 인증
- JWT 발급/갱신/검증 (액세스 1시간, 리프레시 7일)
- 로그인 / 회원가입 / 관리자 로그인
- 비밀번호 BCrypt 암호화
- 자동 토큰 갱신 (Axios 인터셉터, 401 시 재발급)
- PrivateRoute 로 관리자 페이지 보호
- 휴대폰 OTP 엔드포인트 존재 (`/auth/send-code`, `/auth/verify-code`)

### 상품/교재
- 상품 CRUD + 이미지 업로드
- 학년별·과목별·키워드 검색/필터 + 페이지네이션
- 샘플 데이터 17개 (초·중·고 전 학년) 자동 생성
- 관리자 상품 관리 페이지 완전 구현

### 장바구니
- React Context + localStorage 기반 (의도된 설계)
- 담기·수량 변경·삭제·전체 비우기
- 30,000원 이상 무료 배송 계산

### 주문
- 주문 생성 / 조회 / 취소 (회원 & 비회원)
- 상태 워크플로우: PENDING → PAID → SHIPPING → DELIVERED / CANCELED
- 관리자 주문 목록·상태 변경

### 마이페이지
- 프로필 수정 (이름, 이메일, 전화번호, 주소, 비밀번호 변경)
- 주문 내역 (페이지네이션, 상태 배지, 아이템 상세 펼치기)

### 관리자 페이지 (일부)
- 회원 관리: 목록·검색·삭제·학년 필터
- 주문 관리: 목록·검색·상태 변경
- 상품 관리: CRUD·이미지 업로드
- PG 설정: 공급사별 키·모드·결제수단 설정
- 계좌이체 설정: 은행명·계좌번호·예금주

## 2. 부분 구현된 기능 ⚠️

### 결제 — 백엔드만 구현, 프론트엔드 누락

**백엔드 (완료)**
- PG 클라이언트 팩토리 패턴 (Toss / KakaoPay / PortOne)
- `/payments/confirm` — 결제 금액 검증 후 PG 승인
- Payment 엔티티 (paymentId, orderId, amount, status, pgProvider)

**프론트엔드 (누락)** ← 치명적
- PG SDK 로드 없음 (Toss SDK, 카카오 SDK, PortOne SDK)
- 결제창 호출 코드 없음
- 결제 완료 후 `/payments/confirm` API 호출 없음
- 결제 성공/실패 콜백 처리 없음
- **현재 상황**: 주문 생성 후 결제창이 열리지 않아 PENDING 상태에서 멈춤

### 관리자 대시보드
- 통계 카드·차트 UI 존재하지만 전부 하드코딩된 Mock 데이터
- 실제 API 연결 없음

### 휴대폰 인증
- OTP 전송/검증 엔드포인트 존재
- SMS 발송 서비스 미연결 (콘솔 로그만 출력)
- 회원가입 시 실질적 인증 우회 가능

---

## 3. 미구현 기능 ❌

### 게시판 백엔드
- 프론트 어드민 게시판 페이지(`BoardsPage.tsx`)는 완성
- 백엔드에 Board 엔티티·서비스·컨트롤러 전혀 없음
- 현재 전부 Mock 데이터로 표시

### PG 웹훅
- PG사가 서버에 비동기 결제 결과를 전송하는 웹훅 수신 엔드포인트 없음
- 네트워크 중단 시 결제 상태 누락 위험

### 이메일 알림
- 주문 완료·결제 완료·배송 시작 이메일 없음
- 비밀번호 찾기/재설정 없음

### 재고 관리
- Product 엔티티에 재고(stock) 필드 없음
- 주문 시 재고 차감 없음
- 품절 처리 없음

### 환불/부분취소
- 결제 취소 엔드포인트(`/payments/{id}/cancel`)는 존재
- 부분 취소, 환불 상태 추적, 취소 사유 관리 없음

## 4. 보안 이슈

| 수준 | 항목 | 설명 |
|------|------|------|
| 🔴 높음 | JWT Secret 노출 | `application.properties`에 기본값 하드코딩. 운영 시 환경변수 `JWT_SECRET` 반드시 설정 필요 |
| 🔴 높음 | PG 키 노출 | `pg.toss.secret-key` 등 properties에 평문 저장. 환경변수로 분리 필요 |
| 🟠 중간 | OTP Rate Limit 없음 | 인증 코드 무한 요청 가능. 브루트포스 취약 |
| 🟠 중간 | HTTPS 강제화 없음 | HTTP → HTTPS 리다이렉트 미설정 |
| 🟡 낮음 | 소프트 삭제 없음 | 사용자·상품 하드 삭제, 감사 로그 없음 |
| 🟡 낮음 | 체크아웃 미보호 | `/checkout` 라우트에 인증 가드 없음 (비회원 주문은 의도적) |

## 5. 데이터베이스

- **현재**: H2 인메모리 (서버 재시작 시 데이터 소멸)
- **운영 필요**: PostgreSQL 또는 MySQL
- Flyway/Liquibase 등 마이그레이션 도구 없음

체크아웃 → 실제 PG 결제창 → 결제 완료 처리

```
1. Toss Payments SDK 로드 (index.html 또는 동적 스크립트)

```
application.properties에서 H2 → PostgreSQL/MySQL로 변경

## 7. 현재 동작 가능한 테스트 시나리오

서비스를 지금 띄웠을 때 아래 흐름은 **정상 동작**합니다:

```
✅ 회원가입 → 로그인 → 교재 검색 → 상품 상세 → 장바구니 담기
✅ 장바구니 → 수량 변경 → 주문서 작성 → 주문 생성 (PENDING)
✅ 마이페이지 → 주문 내역 조회 → 주문 취소
✅ 관리자 로그인 → 상품 등록/수정/삭제 → 회원 관리 → 주문 상태 변경
✅ 무통장입금 선택 시 계좌 안내 (PG 결제와 달리 수동 확인 방식으로 운영 가능)
```

아래 흐름은 **현재 불가**:

```
❌ PG 카드결제 / 카카오페이 / 토스페이 실제 결제
❌ 게시판 실제 데이터 표시 (모두 Mock)
❌ 관리자 대시보드 실제 통계
❌ 휴대폰 OTP 실제 발송
```

## 8. 무통장입금으로 임시 운영하는 경우

PG 결제 없이 무통장입금만으로 소규모 운영을 시작하려면:

1. 관리자 설정 → 계좌이체 설정에 계좌 정보 입력
2. CheckoutPage에서 '무통장입금' 선택 강제화 또는 PG 버튼 비활성화
3. 주문 후 관리자가 입금 확인 → 수동으로 주문 상태 PAID 변경
4. 배송 처리 시 SHIPPING → DELIVERED 변경

이 방식은 지금 바로 운영 가능하나, PG 결제 구현 전까지의 임시방편으로만 사용 권장.
